Redis 面试题

发表于2023-03-27|更新于2023-03-27
|字数总计:1.9k|阅读时长:5分钟

什么是 Redis

Redis 是一个开源的高性能的键值对数据库，与传统数据库不同的是，Redis 是保存在内存中的，所以读写速度非常快，因此 Redis 被广泛应用于缓存方向，每秒可以处理超过 10 万次读写操作

​

​

应用场景

Redis 适用于缓存、Session 共享、消息队列和计数器等场景

​

​

缓存数据的处理流程

  1. 如果用户请求的数据在缓存中就直接返回
  2. 缓存中不存在的话就看数据库中是否存在
  3. 数据库中存在的话就返回并更新缓存中的数据
  4. 数据库中不存在的话就返回空数据

​

​

Redis 的线程模型

Redis 6.0 前使用单线程方式接收处理用户请求：Redis 在处理客户端的请求时，包括对数据的读取和操作都由一个顺序串行的主线程处理，避免了多线程切换等性能开销

Redis 6.0 后可开启多线程接收处理用户请求，主要是为了提高网络 I/O 的读写性能，充分利用服务器 CPU 资源

​

​

Redis 缓存过期时间

为什么要设置 Redis 缓存过期时间：

  • 因为内存是有限的，如果缓存中的数据一直保存的话，很容易出现内存不足的现象，而且在很多应用场景下，比如短信验证码可能只在 1 分钟内有效，设置数据的缓存过期时间有助于缓解内存的消耗

​

​

数据删除策略

Redis 采用的是定期删除 + 惰性删除机制

  • 定期删除：每隔一段时间抽取一批 key 执行删除过期 key 操作
  • 惰性删除：只有在对数据进行操作时才会执行过期检查，对 CPU 友好，但是可能会造成大量过期的 key 没有被删除

​

​

Redis 数据类型

  • 字符串 string：最基本的 Redis 值类型，能包含任意类型的数据
  • 列表 list：由多个元素组成的有序、可重复的集合
  • 集合 set：由多个元素组成的无序、无重复的集合
  • 有序集合 sorted set：类似于集合，但是每个元素都有一个关联的分数，可以按照分数排序
  • 哈希 hash：由多个键值对组成的无序集合

​

​

慢查询

  • 因为 Redis 中命令执行的排队机制，慢查询会导致其他命令的级联阻塞，可以通过慢查询日志，将命令执行时长超过设定阈值时，将此命令发送的时间、执行时长、执行命令等信息记录下来
  • 慢查询的原因有许多，有可能网络传输有延迟，也有可能排队的命令比较多等原因

​

​

Redis 数据持久化

与传统数据库不同，Redis 是记录在内存中的，因此重启服务等情况都会导致数据的丢失，这个时候就需要将内存中的数据保存到磁盘文件中，实现数据的持久化保存。Redis 支持两种数据持久化的方法：

  • RDB 快照：定时对 Redis 中的数据执行快照，保存到磁盘上的一个快照文件中，RDB 只保留当前最新版本的一个快照，通过该文件可以还原生成该文件时的数据库状态。适用于可以承受部分数据丢失的场景
  • AOF 日志：日志记录，将 Redis 的每个写操作都追加到一个文件中，当 Redis 重启时，它会重新执行 AOF 文件中的所有写操作，以恢复数据。适用于不能容忍数据丢失的场景，但是通常建议同时开启 RDB 和 AOF

​

​

Reids 与 Memcached 对比优势所在

  • Memcached 所有的值均是简单的字符串，Redis 支持更为丰富的数据类型
  • Redis 的速度比 Memcached 快很多
  • Redis 可以实现数据持久化

​

​

Redis 主从复制

Redis 的主从复制架构是一种高可用的解决方案，它可以将一个 Redis 实例的数据复制到多个从节点上，从而实现数据的备份和容错，主节点负责写操作，从节点负责读操作。当主节点出现故障时，可以通过手动或自动切换将从节点升级为主节点，从而实现高可用性的数据服务

  • 主节点将写操作记录到内存中的命令缓冲区中
  • 主节点将命令缓冲区中的写操作同步到从节点
  • 从节点接收到主节点的写操作后，将其记录到自己的内存中
  • 从节点定期向主节点发送 SYNC 命令，请求更新自己的数据
  • 主节点接收到 SYNC 命令后，将自己的数据发送给从节点
  • 从节点接收到主节点的数据后，将其更新到自己的数据中

​

​

全量复制和增量复制

Redis 中的主从复制有全量复制和增量复制

  • 全量复制：在初次复制或从节点重新连接到主节点时，主节点会将所有数据发送给从节点
  • 增量复制：在全量复制完成后，主节点会将自己的写操作缓存到内存中，并发送给从节点

​

​

Redis 哨兵

Redis 哨兵是 Redis 的一个高可用性解决方案，用于监控 Redis 主节点和从节点的状态，并在主节点宕机时自动将从节点切换为主节点，从而保证 Redis 集群的高可用性

​

​

Redis 哨兵工作原理

Redis 哨兵通过向 Redis 主节点和从节点发送命令和心跳包来监控它们的状态，如果主节点宕机或出现故障，哨兵会自动将从节点切换为主节点，并通知其他哨兵和客户端更新集群状态

​

​

Redis 哨兵故障转移策略

Redis 哨兵的故障转移策略包括三个阶段：主观下线、客观下线和故障转移

当哨兵节点检测到主节点宕机时，会将主节点标记为主观下线状态，如果其他哨兵节点也检测到主节点宕机，则将主节点标记为客观下线状态。当主节点被标记为客观下线状态一定时间后，哨兵节点会进行故障转移，将从节点切换为主节点，并通知其他哨兵和客户端更新集群状态

​

​

Redis 集群

使用哨兵只能解决 Redis 高可用问题，实现 Redis 的自动故障转移，但仍然无法解决 Redis Master 单节点的性能瓶颈问题。Redis 集群是 Redis 官方提供的一种分布式解决方案，它可以将数据分散到多个节点上，从而提高 Redis 的可伸缩性和可用性

​

​

Redis 集群工作原理

Redis 集群采用分片的方式将数据分散到多个节点上，每个节点都负责一部分数据。客户端向集群发送请求时，集群会根据请求的 key 值计算出对应的节点，并将请求转发到该节点上

​

​

如何保证 Redis 集群数据的一致性

Redis 集群采用一致性哈希算法来分配数据，每个节点都负责一部分数据。当集群中某个节点故障时，集群会将该节点负责的数据迁移到其他节点上，从而保证数据的一致性
