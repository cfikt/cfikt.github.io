Redis 哨兵

背景

Redis 主从复制架构无法实现 Master 和 Slave 之间角色的自动切换（即当 Master 出现故障时，不能实现自动将 Slave 提升为 Master），主从复制无法实现故障自动转移功能

​

Redis 哨兵（Sentinel）解决了这一问题：实现 Master 和 Slave 角色的自动切换，并且不影响业务

​

架构

Sentinel 实现原理：

  • 有一个专门的 Sentinel 服务进程用于监控 Redis 集群中 Master 的工作状态，当 Master 服务器发生故障时，实现 Master 和 Slave 角色的自动切换，从而实现系统的高可用性
  • Sentinel 是一个分布式系统，需要在多个节点上各自同时运行一个 Sentinel 进程，Sentinel 进程通过留言协议（gossip protocols）来接收 Master 状态（是否下线），使用投票协议（Agreement Protocols）来决定是否执行自动故障转移（少数服从多数），并选择合适的 Slave 作为新的 Master

注意：

  1. Redis Sentinel 中的 Sentinel 节点个数应该为 大于等于 3 且最好为奇数
  2. 客户端初始化时连接的是 Sentinel 节点集合，不再是具体的 Redis 节点，即 Sentinel 只是配置中心不是代理
  3. Redis Sentinel 集群依赖于主从复制架构的实现
  4. Redis Sentinel 节点与普通 Redis 实例没有区别
  5. Sentinel 机制类似于 MySQL 中的 MHA 功能，只解决 Master 和 Slave 角色的自动故障转移问题，但单个 Master 的性能瓶颈问题并没有解决

​

工作机制

  1. 每个 Sentinel 进程会向其它 Sentinel、Master、Slave 定时发送消息，来确认对方是否存活，如果发现某个节点在指定配置时间内未得到响应，则将此节点判断为离线，即为主观宕机（Subjective Down，简称为 SDOWN）， —— 我自己觉得你 DOWN了
  2. 如果哨兵集群中的多数 Sentinel 进程认为 Master 存在 SDOWN，共同利用 is-master-down-by-addr 命令互相通知后，则认为客观宕机（Objectively Down， 简称 ODOWN）， —— 我们都觉得（一半以上）你 DOWN了
  3. 接下来利用投票算法，从所有 Slave 节点中，选一台合适的 Slave 将之提升为新 Master 节点，然后自动修改其它 Slave 相关配置，指向新的 Master节点，最终实现故障转移 failover

Redis 集群

背景

使用哨兵 Sentinel 只能解决 Redis 的高可用问题，实现 Redis 的自动故障转移。仍然无法解决 Redis Master 单节点的性能瓶颈问题

  • 为了解决单机性能的瓶颈，提高 Redis 服务整体性能，可以使用分布式集群的解决方案
  • Redis 无中心架构的 Redis Cluster ，支持多个 Master 节点并行写入和故障的自动转移动能

在 Redis Cluster 中，读写操作都发生在主节点上，从节点不可读不可写，只发挥故障转移效果

​

集群架构

Redis Cluster 需要至少 3个 Master 节点才能实现，Slave 节点数量不限，通常情况下一般每个 Master 都至少对应的有一个 Slave 节点

LVS 负载均衡策略

静态方法

仅根据算法本身进行调度

  1. RR（Round Robin）：轮询，按照顺序逐一分配到不同的 RS 上；较常用
  2. WRR（Weighted RR）：加权轮询，通过配置权重，指定轮询几率，权重越大，访问几率越大；较常用
  3. SH（Source Hashing）：原地址Hash，将来自同一个 IP 地址的请求始终发往第一次选中的 RS 上；实现会话绑定
  4. DH（Destination Hashing）：目标地址Hash，第一次轮询调度至 RS，后续将发往同一个 IP 地址的请求始终转发至第一次选中的 RS 上；实现 Web 缓存

​

动态方法

根据每个 RS 当前的负载状态及调度算法进行调度：Overhead = Value（即负载值），值较小的 RS 优先调度

  1. LC（Least Connections）：最小连接数；把新的连接请求发送到当前连接数最小的 RS，适用于长连接应用，较常用

  2. WLC（Weighted LC）：加权最小连接数；第一次随机连接，后续权重越大，优先级越高，默认调度方法，较常用

  3. SED（Shortest Expection Delay）：最短期望延迟调度；初始连接高权重优先，后续基于权重分配；基于 WCL 算法，只检查活动连接，而不考虑非活动连接

  4. NQ（Never Queue）：永不排队/最少队列调度；第一轮均匀分配，后续 SED

  5. LBLC（Locality-Based LC）： 基于地址的最小连接数调度，动态 DH，即动态目标地址Hash；根据后端服务器负载动态调度，使用场景：根据负载状态实现正向代理，实现Web Cache等

  6. LBLCR（LBLC with Replication）：基于地址带重复最小连接数调度，带复制功能的 LBLC，解决LBLC负载不均衡问题

​

新增调度算法

  • FO（Weighted Fail Over）：静态算法；遍历虚拟服务所关联的真实服务器链表，找到还未过载（未设置 IP_VS_DEST_F_OVERLOAD 标志）且权重最高的真实服务器，进行调度

  • OVF（Overflow-connection）：动态算法；基于真实服务器的活动连接数量和权重值实现，遍历虚拟服务相关联的真实服务器链表，找到权重值最高的可用真实服务器，将新连接调度到权重值最高的真实服务器，直到其活动连接数量超过权重值，之后调度到下一个权重值最高的真实服务器

HTTP 通信过程

浏览器访问网站过程

​

客户端发起请求：

  • DNS 解析
    • 先检查自己本地的 hosts 文件是否有这个网址映射关系，若有，则调用该 IP 地址映射，完成域名解析
    • 若没有，则查询本地 DNS 缓存，若命中缓存，则直接返回结果
    • 若缓存不存在，则向根服务器进行迭代查询
  • Socket 建立连接（TCP/IP 建立连接），三次握手
  • Web 服务器处理 HTTP 请求
    • 建立连接：接收或拒绝连接请求
    • 接收请求
    • 处理请求：服务器对请求报文进行解析，并获取请求的资源及请求方法等相关信息进行处理（GET、POST、PUT、…… ）
    • 获取资源：服务器获取请求报文中请求的资源 Web 服务器（即 Web 服务器向客户端提供资源）
    • 构建响应报文
    • 发送响应报文
    • 记录日志（最后，当事务结束时，Web 服务器会在日志文件中添加一个条目，来描述已执行的事务）
  • 客户端接收响应报文，进行页面渲染
  • 关闭连接

​

HTTP 请求处理过程

  1. 建立连接：接收或拒绝连接请求

​

  1. 接收请求：接收客户端请求报文中对某资源的一次请求的过程

Web 访问响应模型（Web I/O）：

  • 单进程 I/O 模型：启动一个进程处理用户请求，而且一次只处理一个，多个请求被串行响应
  • 多进程 I/O 模型：并行启动多个进程，每个进程响应一个连接请求
  • 复用 I/O 结构：启动一个进程，同时响应多个连接请求
  • 复用的多进程 I/O 模型：启动 M 个进程，每个进程响应 N 个连接请求，同时接收 M * N 个请求

  1. 处理请求：服务器对请求报文进行解析，并获取请求的资源及请求方法等相关信息进行处理（GET、POST、PUT、…… ）

  2. 获取资源：服务器获取请求报文中请求的资源 Web 服务器（即 Web 服务器向客户端提供资源）

  3. 构建响应报文
  4. 发送响应报文
  5. 记录日志（最后，当事务结束时，Web 服务器会在日志文件中添加一个条目，来描述已执行的事务）

Web 相关术语

  • 互联网：Internet，“网络的网络”，泛指有多个计算机网络互连而成的网络，是所有类型网络的母集
  • 因特网：internet，指当前全球最大的、开放的、有众多网络互相连接而成的特定的计算机网络
  • 万维网：www，是文件、图片、多媒体和其他资源的集合，资源通过超链接互相连接形成网络，万维网也称为 Web，是因特网中的一项服务（Web服务）

​

  • URL：统一资源定位符，标志万维网上的各种文档，并使每个文档在整个因特网的范围内具有唯一的标识符 URL
  • HTTP：超文本传输协议，是一种用于分布式、协作式和超媒体信息系统的应用层协议
  • HTML：超文本标记语言，是一种用于创建网页的标准标记语言，复制实现 Web 页面的结构

​

  • IP（Internet Protocol）：独立 IP，记录远程客户机同一 IP 地址访问网站的次数，是衡量网站流量的重要指标，一天内来自相同客户机的 IP 地址只计算一次
  • PV（Page View）：访问量，页面浏览量或点击量，反映的是浏览某网站的页面数（即网站被访问的页面数量），用户每次刷新即被计算一次
  • UV（Unique Visitor）：独立访客，问某网站的电脑数量（通过 Cookies 实现），一天内相同的客户端只被计算一次，如果更换了 IP 但不清除 Cookies，再访问相同网站，该网站的统计中 UV 数是不变的
  • QPS（Request Per Second）：每秒请求数

网络 I/O 模型

阻塞型、非阻塞型、复用型、信号驱动型、异步

​

阻塞型 I/O

  1. 阻塞 I/O 模型是最简单的 I/O 模型，用户线程在内核进行 I/O 操作时被阻塞
  2. 进程发起 I/O 系统调用后，进程被阻塞，转到内核空间处理，整个 I/O 处理完毕后返回进程。操作成功则进程获取到数据
  3. 进程发起 I/O 系统调用后，一直被阻塞，直到数据准备好且从内核复制到用户程序才返回
  4. 使用系统调用，并一直阻塞直到内核将数据准备好，之后再由内核缓冲区复制到用户态，在等待内核准备的这段时间什么也干不了

​

特点：

  • 进程阻塞挂起不消耗 CPU 资源，及时响应每个操作
  • 实现难度低、开发应用较容易
  • 适用并发量小的网络应用开发

不适用并发量大的应用：因为一个请求 I/O 会阻塞进程，所以，得为每请求分配一个处理进程（线程）以及时响应，系统开销大

​

非阻塞型 I/O

进程发起 I/O 系统调用后：

  • 如果内核缓冲区没有数据，内核会返回错误码，进程不再等待，继续处理其他请求，并且隔一段时间不断轮询询问内核数据是否准备好
  • 如果内核缓冲区有数据，内核就会把数据返回进程

​

特点：

  • 进程轮询（重复）调用，消耗 CPU 的资源
  • 实现难度低、开发应用相对阻塞 I/O 模式较难
  • 适用并发量较小、且不需要及时响应的网络应用开发

阻塞式 I/O 的轮询会耗费大量 CPU，通常在专门提供某一功能的系统中才会使用

​

多路复用型 I/O

  • 多路复用 I/O 指一个线程可以同时（实际是交替实现，即并发完成）监控和处理多个文件描述符对应各自的 I/O ，即复用同一个线程
  • 一个线程之所以能实现同时处理多个 I/O ，是因为这个线程调用了内核中的 SELECT、POLL 或 EPOLL 等系统调用，从而实现多路复用 I/O
  • 类似与非阻塞，只不过轮询不是由用户线程去执行，而是由内核（代理）去轮询，内核监听程序监听到数据准备好后，调用内核函数复制数据到用户态

​

特点：

  • 专一进程解决多个进程 I/O的阻塞问题，性能好；Reactor 模式（反应器模式）
  • 实现、开发应用难度较大；
  • 适用高并发服务应用开发：一个进程（线程）响应多个请求

​

多路复用实现方式：

  • SELECT：线性扫描所有监听的文件描述符，不管他们是不是活跃的；有最大数量限制（Linux 和 Windows）
  • POLL：与 SELECT 类似，不过数据结构不同，没有最大数量限制（Linux）
  • EPOLL：采用主动通知性机制，不需要主动轮询，，没有最大数量限制不会随之 I/O 时间的增长导致性能下降（Linux），是 Nginx 中最高性能的事件驱动库之一

​

信号驱动式 I/O

信号驱动 I/O：进程不用等待（即不阻塞），也不用去轮询。而是让内核在数据就绪时，发送信号通知进程

​

特点：回调机制，实现、开发应用难度大

​

异步型 I/O

异步 I/O 才是真正意义上的非阻塞：

  • 当用户进程发起系统调用后，就立刻可以去做其它的事
  • 当进程发起一个 I/O 操作，进程返回（不阻塞）一个 I/O 处理结果
  • 如果 I/O 操作成功则进程直接获取到数据

​

特点：

  • 不阻塞，数据一步到位；Proactor 模式（关注的不是就绪事件，而是完成事件）
  • 需要操作系统的底层支持
  • 实现、开发应用难度大
  • 非常适合高性能高并发应用

​

五种 I/O 对比

阻塞程度（效率从低到高）：阻塞IO > 非阻塞IO > 多路转接IO > 信号驱动IO > 异步IO

​

​

系统 I/O 模型

同步/异步

同步（synchronous）/异步（asynchronous）：关注事件处理的消息通信机制

同步：被调用者不提供消息返回，调用者主动询问事件是否处理完成

异步：被调用者通过状态、通知或回调机制主动通知调用者

同步：进程发出调用请求后，内核不提供通知机制，即文件 I/O 处理完成后不通知进程，需要进程主动向内核询问是否处理完成

异步：进程发出调用请求后，内核会在处理完成后返回调用结果给进程

​

阻塞/非阻塞

阻塞（blocking）/非阻塞（noblocking）：关注调⽤用者在等待结果返回之前所处的状态

阻塞：I/O 操作彻底处理完成才返回到用户空间，调用结果返回到用户空间之前，调用者被挂起，处于等待状态

非阻塞：I/O 操作被调用后立即返回给用户一个状态值，无需等待 I/O 操作彻底完成，调用结果返回到用户空间之前，调用者不会被挂起，可以执行其它事件
