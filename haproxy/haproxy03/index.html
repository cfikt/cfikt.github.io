HAProxy 调度算法

发表于2023-03-27|更新于2023-03-27|HAProxy 负载均衡
|字数总计:2.4k|阅读时长:9分钟

Socat 工具

socat 工具可以对服务器动态权重和其它状态进行调整

​

  • 查看 HAProxy 后端服务器权重
  • 修改 HAProxy 后端服务器权重

  • 禁用 HAProxy 后端服务器

    平滑修改，即禁用后不会影响已连接用户的请求

​

​

​

​

HAProxy 静态算法

静态算法：按照事先定义好的规则轮询进行调度，不关注后端服务器的工作负载、连接数和响应速度等，且无法通过 socat 工具实时动态修改权重（只能为 0% 和 100%，不支持其它值），若想修改为其它值，修改后只能通过重启 HAProxy 生效

​

​

static-rr 算法

static-rr（Static Round Robin）：基于权重的轮询调度，将新的连接根据权重按照顺序分配给每个后端服务器

HAProxy 将使用 static-rr 调度算法将网络流量分配给两个服务器（server1, server2）。并将新的连接根据权重按照顺序分配给每个后端服务器

​

​

first 算法

first（先来先服务算法）：根据服务器在列表中的位置，自上而下进行调度。当第一台服务器连接数达到上限时，新请求才会分配给下一台服务器

HAProxy 将使用 first 调度算法将网络流量分配给两个服务器（server1, server2）。将连接请求分配到 server1 上，当 server1 的并发连接数达到 100 后，才将新的请求往 server2 上分配

​

​

​

​

HAProxy 动态算法

动态算法：基于后端服务器状态进行适当调整，新请求将优先调度至当前负载较低的服务器，且权重可以在 HAProxy 运行时动态调整，无需重启

​

​

roundrobin 算法

roundrobin（轮询调度算法）

  • 基于权重的轮询动态调度算法，将新的连接根据权重按照顺序分配给每个后端服务器，支持权重的运行时调整。HAProxy 默认的调度算法
  • HAProxy 中的 roundrobin 支持慢启动（新加的服务器会逐渐增加转发数），其每个 backend 中最多支持 4095 个后端服务器

​

支持权重的动态调整

​

​

leastconn 算法

leastconn（加权的最少连接调度算法）：将请求发送到当前连接数最少的服务器上，以平衡服务器负载，支持权重的运行时调整和慢启动。适合长连接的场景使用，比如：MySQL、Redis 等场景

​

​

random 算法

random：根据权重随机选择一个服务器进行请求转发。服务器的权重值越高，它被选中的概率就越大，当有请求到达 HAProxy 时，从可用的服务器列表中随机选择一个服务器，根据该服务器的权重值来计算选择该服务器的概率

HAProxy 会使用 “random” 算法从三个服务器中随机选择一个进行请求转发。其中，服务器 server1 的权重值最高，所以它被选中的概率最大，而服务器 server3 的权重值最低，被选中的概率最小

​

​

​

​

HAProxy 其它算法

其它算法即可作为静态算法，又可以通过选项成为动态算法

​

​

Source 算法

Source 算法：根据客户端的 IP 以及端口号进行哈希计算，并将请求转发到后端服务器，后续同一个源地址请求将被转发至同一个后端服务器，确保同一客户端的所有请求都被分配到同一后端服务器上，从而实现会话保持和负载均衡

​

哈希调度算法会根据客户端 IP 地址的哈希值将请求分配给服务器，这样相同的客户端 IP 地址将总是被分配到同一个服务器上。这种算法适用于需要保持会话状态的应用程序，例如 Web 应用程序，但可能会导致某些服务器负载过高

​

Source 算法默认为 map-base 取模法，为静态方式，可以通过 hash-type 选项更改为一致性哈希

​

​

map-base 取模

根据客户端的 IP 以及端口号进行哈希，再基于后端服务器的总权重取模，而后 HAProxy 会将哈希值映射到后端服务器。如果该服务器处于活动状态，则请求将被发送到该服务器

  • 此方法为静态方法，不支持在线调整权重，不支持慢启动
  • 缺点是当服务器的总权重发生变化时（有服务器上线或下线），都会因总权重发生变化而导致调度结果整体改变

​

一致性 Hash

一致性哈希与 map-base 取模法类似，但是一致性哈希算法可以确保在添加或删除服务器时，只有少量的请求需要重新映射，从而减少了哈希冲突和服务器负载不均的问题

  • 此方法为动态方法，支持使用 socat 等工具进行在线权重调整，支持慢启动
  • 当服务器的总权重发生变化时，对调度结果影响是局部的，不会引起大的变动

​

​

uri 算法

基于用户请求的 URI 做哈希运算，再将哈希结果对总权重进行取模后，根据哈希值将请求分配给后端服务器，访问同一个 URI 将调度到同一个后端服务器上。适用后端为缓存服务器场景

uri 算法默认为 map-base 取模法，为静态方式，可以通过 hash-type 选项更改为一致性哈希

​

map-base 取模

​

​

url_param 算法

基于用户请求 URI 参数部分中的一个参数 Key 与 Value 值做哈希计算，再将哈希结果对总权重进行取模后，根据哈希值将请求分配给后端服务器

url_param 算法默认为 map-base 取模法，为静态方式，可以通过 hash-type 选项更改为一致性哈希

​

​

​

hdr 算法

基于用户请求的主机头部（Header）中的指定信息做哈希计算，再将哈希结果对总权重进行取模后，根据哈希值将请求分配给后端服务器

hdr 算法默认为 map-base 取模法，为静态方式，可以通过 hash-type 选项更改为一致性哈希

​

访问测试：在 HAProxy 配置文件中配置基于用户请求中主机头的 Host 信息做哈希计算。实现访问不同域名调度到不同的后端服务器上

​

算法总结

生产环境中较多数场景下使用 HAProxy 默认的调度算法：roundrobin 基于权重的轮询动态调度算法。uri、url_param、hdr 算法也相对较为常用
