Kubernetes 应用编排

编排过程

在 Kubernetes 中不建议也不应该直接使用 Pod，而是使用一个应用编排控制器来管理 Pod。使用 Kubernetes 编排运行一个应用的核心过程如下：

  1. 选择一个合适的应用编排控制器资源对象（定义一个资源配置文件，并将其提交给 API-Server）
  2. 对于一个服务端应用，通常应该提供一个 Service（提供固定的访问入口及负载均衡）
  3. 对于需要存储数据的应用，还应该为其提供持久化存储数据的 Volume
  4. 对于提供独特配置信息的应用（配置文件），还需要为其配置 ConfigMap（提供配置文件） 和 Secret（注入敏感数据）

​

资源配置文件

在 Kubernetes 系统中，运行或管理一个资源对象，通常需要定义一个资源配置文件，并将其提交给 API-Server。API-Server 是一个声明式 API，提供的是一个编程接口，所以使用 Kubernetes 的前提是编写遵循其 API 资源规范的配置文件

​

Kubernetes 中大多数资源类型的 API 格式相近似，具有五个一级字段，遵循同一个格式：

  1. apiVersion：资源类型所属的群组以及版本号
  2. kind：资源类型本身的名称。例如 Pod、Service 等

  3. metadata：对象元数据。对象的名称、描述等

    1. name：对象名称，在其所属的名称空间必须唯一
    2. namespace：对象所属的名称空间
    3. labels：对象标签,key:value。K8s 系统会对某些资源类型自行维护一至多个 Labels
    4. ……

  4. spec：期望的状态

  5. status：实际的状态。由 Kubernetes 系统控制平面相关组件负责

​

为了便于管理，根据其所属的功能类别，API 资源类型被分成了多个小组，称之为 APIGroup，每个组只包含一部分资源类型，每个组独立进行版本标识

​

API 资源对象管理

API 资源对象管理：使用客户端程序，对 API-Server 的 REST 服务端点发起请求，即对每个资源对象对应的端点发起增删改查操作请求。Kubernetes 系统中 kubectl 命令提供了三种类型的对象管理机制：

  • 指令式命令：使用 kubectl 命令行选项完成对象管理操作，直接作用于集群上的活动对象，适用于执行一次性任务

  • 指令式对象配置：基于配置文件执行对象管理操作，直接指明操作，从配置文件中读取要操作的对象以及对象属性。只能独立引用每个配置清单文件

  • 声明式对象配置：声明对象终态，基于配置文件执行对象管理操作，具有幂等性。可直接引用目录下的所有配置清单文件，也可直接作用于单个配置文件

​

​

部署并访问应用

部署应用：

  1. 依照编排需求，选定合适类型的工作负载类型控制器
  2. 创建工作负载控制器对象，由其确保运行合适数量的 Pod 对象
  3. 创建 Service 对象，为该组 Pod 对象提供固定的访问接口

​
