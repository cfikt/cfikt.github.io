持久卷

持久卷是一种集群级别的存储卷，它独立于 Pod，可以被多个 Pod 使用，与数据卷不同，持久卷通常用于持久化存储数据，以确保在 Pod 重新调度或删除时数据不会丢失。持久卷的存储通常由集群管理员创建和配置，然后由开发人员在 Pod 的 spec 中使用

​

持久卷相关概念：

  • PV（持久卷，Persistent Volume）：集群中的一块存储，是对存储资源的抽象，将后端存储定义为一种容器可以使用的资源，由管理员定义。集群级别的资源，生命周期独立于 Pod 之外

    某个存储服务的抽象，对应于一种 Kubernetes 的标准资源类型，通过资源定义文件中的配置参数对接到某个后端存储服务上

  • PVC（持久卷声明，Persistent Volume Claim）：用户对持久卷存储的请求，用于在空闲的 PV 中申请使用符合过滤条件的 PV，由用户定义。名称空间级别的资源，与选定的 PV是 “一对一” 的关系

    PV 的过滤器，指定一至多个过滤条件，从而在可用的 PV 请求绑定一个 PV，是 Pod 能够请求绑定 PV 的入口

  • 存储类（StorageClass）：动态生成 PV 的模板，在用户申请持久卷存储请求的时候，自动为其生成 PV

静态 PV 与 PVC

PV 是集群中的一块存储，管理员将存储资源抽象为 PV，用户通过 PVC 对 PV 进行存储的请求，PVC 消耗 PV，与选定的 PV是 “一对一” 的关系。使用静态 PV 和 PVC 的配置卷步骤：

  1. 创建持久卷：在集群中创建一个 PersistentVolume，它可以使用多种不同的存储后端（如 NFS、AWS EBS、Azure Disk 等）
  2. 定义持久卷声明：在集群名称空间级别中定义一个 PersistentVolumeClaim，该声明描述了需要使用的存储大小、存储类别等信息
  3. 挂载持久卷：在 Pod 的 spec 中使用 PersistentVolumeClaim.claimName 字段指定使用哪个 PersistentVolumeClaim，将持久卷挂载到容器中

​

PV 资源定义

PV 是标准的资源类型，除了负责关联后端存储系统，还需要定义支持的存储特性：

资源字段 描述
.kind PersistentVolume
.spec .capacity.storage 设置持久卷存储容量
.volumeMode 当前 PV 卷提供的存储空间模型：
- Filesystem：文件系统（默认值）
- BlockDevice：块设备
accessMode 支持的访问模型：
- ReadWriteOnce：单路读写
- ReadWriteMany：多路读写
- ReadOnlyMany：多路只读
- ReadWriteOncePod：只能被一个 Pod 以读写模式挂载
.persistentVolumeReclaimPolicy 删除 PVC 时，PV 的回收策略：
- Retain：保留
- Delete：删除

​

NFS PV 资源定义范例：

STATUS：

  1. Available（可用）：一块空闲资源还没有被任何声明绑定
  2. Bound（已绑定）：卷已经被声明绑定
  3. Released（已释放）：声明被删除，但是资源还未被集群重新声明
  4. Failed（失败）：该卷的自动回收失败

存储类

存储类（Storage Class）：Kubernetes 支持的标准资源类型。是动态生成持久卷的模板，也是 PVC 筛选 PV 时的过滤条件之一

​

若想使用存储类动态预置 PV，需要存储类与后端存储服务建立关联关系，而建立关联关系的前提是后端存储服务需要支持管理 API。有些存储服务默认不支持动态 PV 机制（即不支持该 API），可以为其添加 CSI 接口来实现对动态 PV 的支持

​

​

存储扩展知识

存储卷的具体管理操作是由相关的控制器向卷插件发起调用请求来完成的：

在 Kubernetes 系统中，控制器管理器内部具有两个重要的控制器 AD Controller 以及 PV Controller，而存储卷管理器存在于每个 Kubelet 之上。控制器通过 Kubelet 借助卷插件指挥存储卷管理器完成附加、移除、挂载、卸载等操作

​

通过 CSI 规范扩展树外卷插件：

  • CSI Controller：负责与存储服务的 API 通信，从而完成后端存储的管理操作。由 StatefulSet 控制器对象编排运行，副本数为 1，以保证该存储服务只会运行单个 CSI Controller 实例
  • Node Plugin：也称为 CSI Node。负责在节点级别完成存储卷的管理。由 DaemonSet 控制器对象编排运行，以确保每个节点上精确运行一个 Node Plugin 相关的 Pod 副本
