Service 服务

Service 原理

Service：Kubernetes 中的 Service 是一种抽象，它定义了一组逻辑上相关的 Pod 的访问策略，将其对外暴露提供固定的访问接口并提供服务，并实现负载均衡机制。它把 Kubernetes 集群中的所有节点，都作为所有 Service 的负载均衡器，即任何一个节点都可以作为访问集群上某一个服务的入口

​

​

Service 构成

Kubernetes 中的 Service 是一种抽象，它定义了一组逻辑上相关的 Pod 的访问方式，使得这些 Pod 能够被其他服务或者用户轻松地访问。Service 主要由以下三个组成部分构成：

  1. Label Selector：标签选择器。Service 通过标签选择器来选择过滤一组相应的 Pod，这些 Pod 将会被 Service 管理，在逻辑上实现一组 Pod 的分类
  2. Virtual IP：虚拟 IP。Service 会为它所管理的每一组 Pod 分配一个虚拟 IP，当一个客户端访问这个虚拟 IP 时，它会被负载均衡到对应的 Pod 上。需要说明的是，这个虚拟 IP 与 Pod 实际的 IP 没有任何关系
  3. Port Mapping：端口映射。Service 定义了一组端口，这些端口会映射到 Pod 的实际端口上。当一个客户端访问 Service 的端口时，它会被负载均衡到对应的 Pod 的端口上

​

​

创建过程

  1. 用户通过 Kubernetes API 创建 Service 对象，指定 Service 的名称、命名空间、端口等信息
  2. Kubernetes API Server 接收到创建 Service 的请求后，会将 Service 对象的定义信息存储到 ETCD 数据库中
  3. Service Controller 监听 ETCD 数据库中 Service 对象的变化，并根据变化来执行相应的操作。Service Controller 主要负责管理 Service 对象的生命周期，即根据用户创建、更新、删除等操作，调用 API Server 的接口进行相应的处理，确保 Service 的状态与期望一致
  4. 当 Service 对象被创建时，Service Controller 会根据 Service 的定义信息生成一组 Endpoints 对象，并将这些对象存储到 ETCD 数据库中。这些 Endpoints 对象用于存储 Service 所选择的 Pod 的 IP 地址和端口号信息
  5. Kube-Proxy 监听 ETCD 数据库中与 Service 和 Endpoint 相关的变化。当 Service Controller 创建新的 Endpoints 对象时，Kube-Proxy 会根据 Service 的定义信息和 Endpoints 对象中存储的 Pod IP 地址和端口信息，生成相应的 iptables 或者 ipvs 规则，并将这些规则应用到本地的节点上
  6. Service 和 Pod 之间的网络流量经过 Kube-Proxy 生成的 iptables 或者 ipvs 规则进行负载均衡和转发，以实现服务的访问

需要注意的是，以上步骤中的 Service Controller 和 Kube-Proxy 都是 Kubernetes 中的核心组件，它们分别负责 Service 和 Endpoint 的管理、Pod 的选择和负载均衡等功能。用户不需要直接操作这些组件，只需要使用 Kubernetes API 创建和管理 Service 对象即可

​

​

工作原理

当 Service 被创建时，Kubernetes 会为它分配一个唯一的 Cluster IP，这个 IP 只能在集群内部使用。当客户端通过 Service 的 Cluster IP 和端口访问 Service 时，请求会被 Service 代理到后端的 Pod 上。这个过程中，Kubernetes 会使用 iptables 或者 ipvs 等技术进行负载均衡，确保请求被均衡地分发到后端的 Pod 上

​

当后端的 Pod 发生变化，比如 Pod 被删除或者新增时，Service 会自动更新它所管理的 Pod 列表，确保客户端可以访问到最新的后端服务。这种自动感知和更新的特性可以使得 Kubernetes 中的服务具备更高的可靠性和可扩展性

​

具体实现原理：

  1. 当用户创建或修改一个 Service 时，这些操作会被发送到 API Server，API Server 会将这些对象的定义存储在 ETCD 中，并通知 Kube-Proxy 监听 Service 对象的变化
  2. 当 Service 对象发生变化时，API Server 会通知 Kube-Proxy 监听这些变化，并触发 Kube-Proxy 进行相应的更新
  3. API Server 还负责对用户的身份进行认证和授权，在进行 Service 对象的创建、更新和删除等操作时，用户需要通过 API Server 进行身份认证，并且只有被授权的用户才能进行这些操作

​

​

工作模式

Service 模式（为集群上的每个 Service 提供负载均衡解决方案）：在淘汰了 UserMode 模式之后，现如今的 Service 主要有 iptables 和 ipvs 两种模式，它们都是基于 Linux 内核中的 Netfilter 实现的

  1. iptables：iptables 是 Linux 内核中提供的防火墙服务。在 Kubernetes 中，每一个 Service 都会生成相应的 iptables 规则，集群中的服务越多 iptables 规则就越多，而 iptables 是从上到下匹配的，因此，当集群服务达到一定的规模时，iptables 模式的效率就会越发底下
  2. ipvs：ipvs 是一个用于负载均衡的 Linux 内核功能。ipvs 采用的时 Hash 算法，并且 ipvs 生成的规程相对于 iptables 来说会少一个数量级，而 ipvs 本身的设计就是用来为大量服务进行负载均衡的，它具有优化的查找算法，而不是简单的从列表中查找规则，因此它的处理效率与集群规模无关，其同步规则的效率和网络吞吐量也更高

​

​

规则转化过程

在 Kubernetes 中，Service 是通过 Kube-Proxy 组件来转化为 iptables 或者 ipvs 规则的。Kube-Proxy 是 Kubernetes 集群中的一个核心组件，它运行在每个节点上，负责为 Service 提供负载均衡和服务发现功能：

  1. 创建 Service：当创建一个 Service 时，Kubernetes 会为这个 Service 分配一个唯一的 Cluster IP 地址，并且为 Service 中所选定的 Pod 分配一个对应的 Endpoint IP（端点 IP） 地址和端口号
  2. 配置负载均衡器：Kubernetes 中使用 iptables 或者 ipvs 来实现 Service 的负载均衡功能，因此需要在节点上配置相应的负载均衡规则。Kubernetes 会自动在节点上为 Service 创建一个 iptables 或者 ipvs 链，然后将负载均衡规则添加到这个链中
  3. 更新规则：当 Service 中的 Pod 发生变化时，Kubernetes 会自动更新负载均衡规则。如果有新的 Pod 加入 Service，Kubernetes 会添加一个新的 Endpoint IP 和端口号，并且更新 iptables 或者 ipvs 规则。如果有 Pod 从 Service 中移除，Kubernetes 会将相应的 Endpoint IP 和端口号从规则中删除
  4. 数据包转发：当客户端发送请求到 Service 的 Cluster IP 和端口时，数据包会首先被 iptables 或者 ipvs 规则匹配，并且根据负载均衡策略被转发到对应的 Pod 的 Endpoint IP 和端口上

总的来说，Kubernetes 通过 iptables 或者 ipvs 来实现 Service 的负载均衡和服务发现功能，使用虚拟 IP 和端口号将多个 Pod 组织在一个逻辑组中，并自动更新负载均衡规则以确保客户端可以访问到最新的后端服务

​

Service 类型

在 Kubernetes 集群中，不论是节点网络、Service 网络、Pod 网络都是配置在系统内核中的。节点内核就是三个网络交汇的位置，集群中的服务端以 Pod 的形式工作在 Pod 网络中，我们的目的就是让其它来源的客户端请求到达 Pod 网络中，Service 类型就是为了接入不同来源的客户端请求到达 Pod 网络中

​

Service 类型：

  1. ClusterIP（集群 IP）：ClusterIP 是 Service 的默认类型，Kubernetes 会为这个 Service 分配一个唯一的 ClusterIP 地址，该地址只能在集群内部使用，可以通过该地址访问 Service 所代表的一组 Pod。ClusterIP Service 适合于内部服务之间的通信，如数据库服务、缓存服务等

  2. NodePort（节点端口）：NodePort 类型的 Service 可以将 Service 暴露到集群外部，在为 Service 分配一个 ClusterIP 的同时还会自动为 Service 配置一个固定节点端口 NodePort，节点端口范围在 30000 - 32767，用户也可以自定义 NodrPort，在集群外部可以通过节点的 IP 地址和固定端口访问 Service，需要说明的是节点端口与服务自身的端口没有任何关系。NodePort Service 适合于需要从外部访问的服务，如 Web 服务器、API 服务器等

  3. LoadBalancer（负载均衡器）：NodePort 类型的扩展，将 Service 暴露到云厂商提供的负载均衡器上，从而可以实现集群外部的负载均衡，在 NodePort 之上使用云提供商的负载均衡器向外部暴露服务，外部负载均衡器流量路由到的 NodePort 服务和 ClusterIP 服务是自动创建的。LoadBalancer Service 适合于需要外部负载均衡和流量管理的服务

  4. ExternalName（外部名称）：将集群外部的服务引入到集群内部来，在集群内部直接使用，也可将 Service 映射到集群外部，借助于 ClusterDNS 上的 CNAME 资源记录完成，将 Service 的名称映射到一个 CNAME 记录。无需借助 ClusterIP 和 NodePort，也无须定义标签选择器发现 Pod 对象，只能通过 DNS 解析访问。适合于需要访问外部服务的场景，如数据库、缓存等

    为 Service 定义一个外部名称，当访问 ServiceName 时，会解析为外部名称，再将外部名称解析为 IP 地址，而后 Pod 才能与这个 IP 地址通信

Service 资源定义

创建一个 Service 资源，主要定义如下内容：

  • 负载均衡器入口：ClusterIP 及相关的 Service Port、NodePort（根据通信需求，确定选择的类型）
  • 标签选择器：用于筛选过滤 Pod，并基于筛选出的 Pod 的 IP 地址生成后端端点列表
  • Service 类型的专有配置

Service 名称解析

在成功部署 Kubernetes 集群后，一定有一个 CoreDNS 组件，它能够动态的完成 Service 的名称解析，对集群中的 Service 和 Pod 进行 DNS 命名解析，从而实现集群内部的服务发现和通信。CoreDNS 会自动监视 API Service 上每一个 Serivce 资源的创建和更改，并且从每一个 ServiceName 到 ClusterIP 做 A 记录，从 ClusterIP 到 ServiceName 做 PTR 记录，把 ServicePort 做 SRV 资源记录，从而建立起映射关系

  • A 记录：从 ServiceName 到 ClusterIP
  • PRT 记录：从 ClusterIP 到 ServiceName
  • SRV 记录：为端口名称标识解析为端口号

​

具体来说，当 Pod 通过 DNS 域名访问 Service 时，Kubernetes 会将该域名解析为 Service 的 ClusterIP 地址，从而实现对 Service 的访问；当 Pod 通过 DNS 域名访问其他 Pod 时，Kubernetes 会将该域名解析为对应 Pod 的 IP 地址，从而实现对 Pod 的访问

​

​

ClusterDNS & CoreDNS

ClusterDNs 和 CoreDNS 的关系：

  • ClusterDNS 是 Kubernetes 中内置的一种 DNS 服务发现机制，它通过对集群中的 Service 和 Pod 进行 DNS 命名解析，从而实现集群内部的服务发现和通信
  • ClusterDNS（CoreDNS）是 Kubernetes 集群中的必备附件，CoreNDS 实现了 ClusterDNS 的功能，将 Service 和 Pod 进行 DNS 命名解析，并支持通过插件扩展其他 DNS 功能。因此，CoreDNS 可以视为 ClusterDNS 的升级版，是 Kubernetes 集群内部 DNS 服务的推荐选择

​

ClusterDNS（CoreDNS）是 Kubernetes 集群中的必备附件，负责为 Kubernetes 提供名称解析和服务发现。每个 Service 资源对象，在 CoreDNS 上都会自动生成一个遵循 <service>.<namespace>.svc.<zone> 格式的名称，围绕该名称会生成一些 DNS 格式的资源记录

​

CoreDNS 会持续监视 API Service 上 Service 资源对象的变动，并实时反映到相关的 DNS 记录中。Pod 在由 kubectl 创建时会根据指定的配置，默认在各容器中 /etc/resolv.conf 文件里，将 NameServer 指向 CoreDNS 相关 Service 的 ClusterIP

​

​

DNS 解析策略

Kubernetes 支持在单个 Pod 资源规范上自定义 DNS 的解析策略和配置，并组合生效

  • spec.dnsPolicy：解析策略

  • spec.dnsConfig：名称解析机制

Hedless Service

Service 各类型中，ClusterIP、NodePort、LoadBalancer 都会为 Service 配置一个 ClusterIP，在 CoreDNS 上，这些 Service 对象的 A 记录也会解析为它的 ClusterIP

​

Hedless Service 称为无头服务，“头”指的是 Service 中的 ClusterIP，而无头服务就是不会分配 ClusterIP 的 Service。当我们访问一个 Service 时，Kubernetes 会自动将其解析为该 Service 的 ClusterIP，若没有 ClusterIP 时，将使用无头服务机制实现访问

​

Hedless Service 的两种类型：

  • 使用 ExternalName 类型的 Service，在 Service 的资源配置文件中定义了一个外部名称，此时访问 ServiceName 会将其解析为该外部名称，再将外部名称解析为 IP 地址，完成通信
  • 在 Service 的资源配置文件中将 clusterIP 字段定义为 None，此时访问 ServiceName 会将其解析为 Pod 的 IP 地址

​

​

Service 流量转发

  • 客户端 Pod 访问 ServiceName 时，由 Service 进行负载均衡（源地址：客户端 Pod；目标地址：Service）
  • Service 将请求转发给后端 Pod（源地址：客户端 Pod；目标地址：后端 Pod）
  • 后端 Pod 接收请求，并直接转发给客户端 Pod（源地址：后端 Pod；目标地址：客户端 Pod）
