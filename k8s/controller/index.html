编排控制器

编排控制器简介

Pod 是 Kubernetes 集群中运行和调度应用的最小原子单位，当用户直接创建 Pod 时，会由 Scheduler 调度到某个节点上运行，并由 kubelet 管理该 Pod 的生命周期。当 Pod 被误删除或者 Pod 所在的节点宕机时，Scheduler 不会重新调度该 Pod，该 Pod 也不会进行重建

应用编排核心：精准确保当前名称空间中，有吻合用户期望数量的 Pod 实例在运行

​

Kubernetes Controller 的控制回路：

  • Controller 根据 spec，控制 Kubernetes System 生成 status
  • Controller 持续监控 Kubernetes System 中的 spec 与 status，在每一次控制回路中对二者进行比较，确保 spec 不断趋于或完全等同 status

​

​

工作负载控制器

控制器：以编排 Pod 化运行的应用为核心的控制器，通常称之为工作负载型控制器

  • 无状态应用编排：Deployment、ReplicaSet
  • 有状态应用编排：StatefulSet、第三方的 Operator
  • 系统级应用编排：DaemonSet
  • 作业类应用编排：Job、CronJob

​

Deployment

Deployment 简介

Deployment 控制器的主要职责是保证 Pod 资源对象的健康运行，该控制器的大部分功能是通过调用 ReplicaSet 控制器实现，而 Deployment 主要是为 Pod 提供滚动更新/回滚功能，一个 Deployment 为 Pod 和 ReplicaSet 提供声明式的更新能力

需要注意的是：在 Deployment 资源对象中，Deployment 控制器并不直接管理 Pod，而是通过 Deployment 控制器控制 Deployment 对象创建出一个 RS 对象，RS 控制器控制 RS 对象创建 Pod 对象，而 Deployment 只是在 RS 的基础上提供滚动更新/回滚功能，创建 Deployment 的时候会同步创建出一个 RS，应用编排是通过 RS 实现的，也就是说真正管理 Pod 的是 RS

​

理解概念：

  • Deployment Controller（Deployment 控制器）：Controller-Manager 中以守护进程运行的一个进程，该进程管控所有 Deployment 资源对象
  • Deployment Resource（Deployment 资源）：API Service 中的资源类型。可以创建多个 Deployment 资源对象，每个资源对象都受控于 Deployment 控制器，每个资源对象都会管理编排一个应用（Pod）

​

​

Deployment 资源规范

Deployment 资源配置文件中的核心配置项：① replicas：Pod 数量；② selector：标签选择器；③ template：Pod 模板

DaemonSet

DaemonSet 简介

DaemonSet 与 Deployment 类似，基于标签选择器管控一组 Pod，但是，DaemonSet 用于确保所有或选定的工作节点上都运行有一个 Pod 副本

​

DaemonSet 的主要作用是用来控制 Daemon Pod，Daemon Pod 具有以下特性

  • 这个 Pod 运行在 Kubernetes 集群中的每一个节点（Node）上
  • 每个节点上只能运行一个 Deamon Pod 实例
  • 当有符合条件的新节点（Node）加入到 Kubernetes 集群时，Daemon Pod 会被自动拉起
  • 当有旧节点被删除时，其上运行的 Daemon Pod 也将被删除

​

DaemonSet 常用场景：

  • 在集群每个节点上启动一个存储守护进程，比如 glusted 或者 ceph
  • 在每个节点上启动一个日志收集进程，比如 fluentd 或者 filebeat
  • 在集群的每个节点上面启动监控的守护进程，比如 Prometheus 的 node-exporter

​

​

DaemonSet 特点

DaemonSet 与 Deployment 类似，有所不同的是，DaemonSet 不支持定义 replicas，并且 DaemonSet 中的滚动更新只支持先删除后更新

​

每个节点能且只能运行一个应用实例

每个节点：

  • 集群中的每个节点（默认）
  • 被节点选择器匹配到的节点

滚动更新仅支持先删除后更新

更新策略：

  • 只支持 maxSurgemax，不支持 Unavailable
  • 先删除一组节点上的实例，更新完成后再进行下一组节点

​

​

DaemonSet 资源规范

DaemonSet 资源配置文件配置项：① 没有 replicas 配置项；② 使用 hostNetwork 共享节点网络；③ 使用 hostPID 共享节点进程

StatefulSet

StatefulSet 简介

对于常规应用都可以通过 Deployment 控制器进行编排、管理，但是 Deployment 有一个致命的缺陷就是不能用来管理有状态应用1，因为它认为所代理的后端的 Pod 都是相同的。

​

对于分布式场景下的有状态应用，使用容器需要很多额外的代价，而 StatefulSet 主要是用来编排和管理有状态应用的，但只是为有状态应用提供了一个基础框架：

  • 为每个 Pod 提供唯一标识。依赖于一个专用的 Headless Service 辅助完成
  • 为每个 Pod 提供一个专有的存储（PVC）。依赖于 volumeClaimTemplates

​

​

第三方 Operator

StatefulSet 控制器只是为有状态应用提供了一个基础框架，而应用集群的运维操作步骤，需要由用户自行编排脚本或程序完成，这对绝大多数用户都是很难完成的任务，所以则需要使用专有、功能完备的控制器（Operator）

  • 拆装 Cluster
  • 自动完成扩缩容中的每个必要步骤
  • 备份、恢复

​

上述功能可以通过声明式 API 实现，并依赖于 自定义资源类型 来完成相应功能的支撑，这些专有的资源类型由专有的 Operator 来负责支撑背后的所有功能

​

​

StatefulSet 资源规范

要创建一个 StatefulSet 资源，需要提供三个组件：

  • StatefulSet 对象：也就是 StatefulSet 控制器
  • Headless Service：确保每个 Pod 都有唯一的网络标识
  • PV：可以认为是 Kubernetes 中的存储，声明存储的时候我们只需要声明一个 PVC 即可，为每个 Pod 提供一个专有的存储

​

Job

Job 简介

前面的 Deployment 和 DaemonSet 主要负责编排始终运行的守护进程类应用，而 Job 负责编排运行有结束时间的“一次性”任务2（批处理应用或者作业类进程）。简单来说，所有具有明确结束标志的应用都可以统称为批处理应用

  • 控制器要确保 Pod 内的进程 “正常（成功完成任务）” 退出
  • 非正常退出的 Pod 可以根据需要重启，并在重试一定的次数后终止
  • 有些 Job 是单次任务，也有些 Job 需要运行多次（次数通常固定）
  • 有些任务支持同时创建及并行运行多个 Pod 以加快任务处理速度，Job 控制器还允许用户自定义其并行度

​

​

Job 资源规范

Job 资源同样需要标签选择器和 Pod 模板，但它不需要指定 replicas，而是应该给定 completions，即需要完成的作业次数，默认为 1 次

  • Job 资源会为其 Pod 对象自动添加“job-name=JOB_NAME”和“controller-uid=UID”标签，并使用标签选择器完成对 controller-uid 标签的关联，因此，selector 并非必选字段
  • Job 资源中，Pod 的 RestartPolicy 的取值只能为 Never 或 OnFailure

​

CronJob

CronJob 简介

CronJob 控制器用于管理 Job 资源的运行时间，它允许用户在特定的时间或以指定的间隔运行 Job，类似 Linux 中的周期性任务作业计划（Crontab），用于控制作业运行的时间点及周期性运行的方式：

  • 仅在未来某时间点将指定的作业运行一次
  • 在指定的周期性时间点重复运行指定的作业

​

需要周期性运行的 Job，则由 CronJob 控制器负责编排

  • CronJob 建立在 Job 的功能之上，是更高层级的控制器
  • CronJob 以 Job 控制器完成单批次的任务编排，而后为这种 Job 作业提供需要运行的周期定义

​

​

CronJob 资源规范

在 CronJob 中，通配符“ ? ”和“ * ”的意义相同，它们都表示任何可用的有效值
