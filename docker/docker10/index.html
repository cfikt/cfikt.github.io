Docker 数据管理

Docker 容器的分层

Docker 镜像是分层设计的，而镜像为只读层，通过镜像启动的容器会添加一层可读写的文件系统，用户写入的数据都保存在这一层中

​

容器的数据分层目录：

  • LowerDir: Image 镜像层；即镜像本身，只读层
  • MergedDir: 容器的文件系统，使用 Union FS（联合文件系统）将 lowerdir 和 upperdir 合并完成后给容器使用，最终呈现给用户的统一视图
  • UpperDir: 容器的上层；可读写，容器变化的数据存放在此处
  • WorkDir: 容器在宿主机的工作目录，挂载后内容会被清空，且在使用过程中其内容用户不可见

​

数据持久化

容器中的数据存储在容器层中，会有以下几个问题：

  1. 删除容器时，容器中的数据也随之删除
  2. 宿主机上的其它进程不便于访问容器内部的数据
  3. 对容器内部的数据的 I/O 会经过存储驱动，再达到宿主机，引入了一层间接层，性能会有所下降

​

因此，就需要对容器内部的数据进行持久化保存，在 Dockerfile 中的 VOLUME 实现的时匿名卷，不便于后续管理；因此，可以使用数据卷，将容器中的数据保存到宿主机的指定目录，保证服务的稳定性以及数据的安全性

​

​

数据卷简介

数据卷（Data Volume）实际上就是宿主机上的文件或者目录，可以被容器直接 mount 使用；在生产环境中，需要针对不同类型的服务、不同类型的数据存储要求做相应的规划，最终保证服务的可扩展性、稳定性以及数据的安全性

​

数据卷使用场景

  • 数据库
  • 日志输出文件
  • 静态 Web 页面
  • 服务配置文件
  • 多容器间目录或文件共享

​

数据卷特点

优点：

  • 数据卷可以在多个容器之间共同使用，实现容器之间共享和重用
  • 对数据卷的更改会在所有容器内立即更新
  • 数据卷的数据可以持久保存，即使删除使用使用该容器卷的容器也不影响
  • 在容器里面的写入数据不会影响到镜像本身，即数据卷的变化不会影响镜像的更新

​

缺点：

  • 依赖于宿主机目录，宿主机出问题则会受到影响，当宿主机较多时，不方便统一管理

​

数据卷保存方式

如果要将写入到容器的数据永久保存，则需要将容器中的数据保存到宿主机的指定目录，Docker 的数据类型分为两种：

  • 数据卷（Data Volume）：直接将宿主机目录挂载至容器的指定的目录，推荐使用
  • 数据卷容器（Data Volume Container）：将宿主机的目录挂载至一个专门的容器中，该容器则称为数据卷容器，间接使用宿主机空间，让其他容器通过数据卷容器读写宿主机的数据

​

数据卷分类

使用 docker run 命令启动容器时，可以指定使用数据卷的方式：

  • 指定宿主机目录或文件：指定宿主机的具体路径和容器路径的挂载关系；此方式不会创建数据卷
  • 匿名卷: 不指定数据名称，只指定容器内目录路径充当挂载点，Docker 自动指定宿主机的路径进行挂载，此方式会创建匿名数据卷，Dockerfile 中VOLUME 指令即为这种
  • 命名卷：指定数据卷的名称和容器路径的挂载关系，此方式会创建命名数据卷

​

​

数据卷使用

使用 docker run 命令中的 -v 选项可以实现数据卷

数据卷容器

数据卷容器介绍

将宿主机与容器之间作过数据卷映射的容器，即使用 docker run -v 或者 Dockerfile 中的 VOLUME 指令创建的容器，都可称为数据卷容器。而数据卷容器的实现方式就是其它容器指定并继承该数据卷容器的数据映射关系，实现多个容器之间共享数据

​

​

实现过程：

  1. 创建一个后台运行的数据卷容器，用于提供数据卷，为其他容器提供数据存储服务 — Server
  2. 其它容器继承该数据容器的数据卷 — Client

​

若 Server 宕机，则其它 Client 都会受到影响

​

数据卷容器使用

使用 docker run 命令的 —volumes-from 选项可以指定使用数据卷容器
