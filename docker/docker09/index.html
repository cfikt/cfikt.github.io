Dockerfile 指令详解

发表于2023-01-01|更新于2023-02-27|Docker 容器技术
|字数总计:1.8k|阅读时长:6分钟

FROM

利用 Dockerfile 构建镜像，首先需要有一个基础镜像，在基础镜像之上镜像定制化操作

​

FROM 指令可以指定基础镜像，该指令必须为 Dockerfile 文件中的首行，后续的指令都运行在该基础镜像提供的环境之上。默认情况下，docker build 会在当前宿主机上查找 FROM 指定的基础镜像，当其不存在时，则会自动从镜像仓库拉取该镜像，无需手动拉取

​

如何选择合适的镜像：建议使用官方提供的镜像，可以有效避免恶意镜像

  • 服务类：各服务官方通常都提供了相关的 Docker 镜像，比如：Nginx、Redis、MySQL 等
  • 系统类：使用操作系统镜像可以实现定制化服务，各系统官方也提供了相关的镜像，比如：CentOS、Ubuntu、Rocky 等

WORKDIR

使用 WORKDIR 指令设置容器运行时的工作目录，后续使用的 RUN、CMD、ENTRYPOINT 指令都将运行在该工作目录之上

​

​

RUN

RUN 指令可以指定构建镜像阶段执行的 SHELL 命令

一个 Dockerfile 中可以存在多个 RUN 指令，每个 RUN 指令都会建立一个镜像层，所以尽可能的将其合并为一条指令。比如将 SHELL 命令通过 && 连接在一起，成为一条指令

ADD & COPY

COPY：拷贝宿主机文件或者目录到镜像中

ADD：拷贝文件或者目录到镜像中且自动解压缩，并且可以下载远程文件

​

注意：

  • 路径为 Dockerfile 文件所在目录的相对路径
  • 如果目标目录不存在，则会自动创建

EXPOSE

指定服务端容器需要对外暴露（监听）的端口，以实现容器与外部环境的通信

  • EXPOSE 仅仅是容器打算使用的端口，并不会真正暴露端口，在启动容器时需要使用 -p 或 -P 才能真正实现端口的映射
  • 即使 Dockerfile 没有使用 EXPOSE 指令，仍然可以用 -p 或 -P 暴露容器内程序真正监听的端口

VOLUME

在容器中创建一个可以从本地主机或其他容器挂载的挂载点，一般用来存放数据库和需要保持的数据等，默认会将宿主机上的目录挂载至 VOLUME 指令指定的容器目录。即使容器后期被删除，此宿主机的目录仍会保留，从而实现容器数据的持久保存

该方法实现的是匿名数据卷，无法指定宿主机路径和容器目录的挂载关系

CMD & ENTRYPOINT

在 Dockerfile 中，CMD 与 ENTRYPOINT 指令都可以实现在运行容器阶段时执行 SHELL 指令，关于这两个指令的区别是 Dockerfile 中比较让人头痛的，在生产环境和面试中也是经常遇到的，是 Dockerfile 的重要知识点：

​

  • CMD：只有最后一个生效，CMD 会被 docker run 之后的参数替换
  • ENTRYPOINT：只有最后一个生效，一定会执行，不会被忽略，docker run 之后的参数会被追加

环境一：

  1. 如果 Dockerfile 中有 CMD 指令，而执行 docker run 时没有传递 SHELL 命令，则执行 CMD 命令
  2. 如果 Dockerfile 中有 CMD 指令，而执行 docker run 时传递了 SHELL 命令，则 CMD 命令会被 docker run 传递的命令替换，执行 docker run 传递的 SHELL 命令

​

环境二：

  1. 如果 Dockerfile 中有 ENTRYPOINT 指令，而执行 docker run 时没有传递 SHELL 命令，则执行 ENTRYPOINT 命令
  2. 如果 Dockerfile 中有 ENTRYPOINT 指令，而执行 docker run 时传递了 SHELL 命令，则 docker run 传递的命令将会追加到 ENTRYPOINT 命令的末尾，作为 ENTRYPOINT 命令的参数执行

​

环境三：

  1. 如果 Dockerfile 中同时存在 CMD 和 ENTRYPOINT 指令，那么 CMD 的命令将会追加到 ENTRYPOINT 命令的末尾，作为 ENTRYPOINT 命令的参数执行
  2. 如果 Dockerfile 中同时存在 CMD 和 ENTRYPOINT 指令，并且执行 docker run 时传递了 SHELL 命令，，则 CMD 命令会被 docker run 传递的命令替换，同时 docker run 传递的命令将会追加到 ENTRYPOINT 命令的末尾，作为 ENTRYPOINT 命令的参数执行
